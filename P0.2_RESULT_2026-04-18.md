## 8. Regression Risks and Deferred Work

### Manual tests not run

The session is a code surgery pass; I did not run the app on device or simulator. The items below are expectations based on code review — they need a live-device pass before TestFlight.

- [ ] Captain chat (Arabic) — now always routes through `BrainOrchestrator` → `CloudBrainService` → Gemini. No behavioral change expected since `CaptainViewModel` already used `BrainOrchestrator`; the deleted CIM path was unreachable from chat.
- [ ] Captain chat (English) — same path (Gemini). Unchanged.
- [ ] Kitchen — meal plan generation now routes through `BrainOrchestrator` with `screenContext: .kitchen`. Caveat below.
- [ ] Workout summary notifications — now 100% template (`fallbackMessage`). Previously deterministic for English on-device path anyway; Arabic users lose the LLM-flavored variance but the template copy is zone-aware and was already the fallback.
- [ ] Inactivity notifications — now 100% template (the same 3-tier copy that used to be the "LLM failed" fallback).
- [ ] Background coach nudges — now 100% template via `IraqiCoachTemplates`. Dev nudge (`queueDeveloperTestCoachNudge`) exercises the same path.

### Known regression — Kitchen plan richness

The previous Kitchen flow sent a custom JSON-schema prompt to the Arabic API or Apple on-device model. Output was expected as a direct JSON with 9 fields per meal including `ingredients`, `protein`, `carbs`, `fat`, `fiber`.

The new flow routes through `BrainOrchestrator`, whose system prompt forces Gemini to respond inside the Captain-shape envelope `{message, quickReplies, workoutPlan, mealPlan, spotifyRecommendation}`. The structured `mealPlan` field Gemini populates has only `{type, description, calories}` per meal (the Captain-level `MealPlan` type, not Kitchen's richer `KitchenMealPlan`).

Parsing order in `KitchenPlanGenerationService.generatePlan`:
1. Try to parse the Kitchen-schema JSON directly out of `reply.message` (`parsePlan`). If Gemini inlined the schema in the message field, this works.
2. Else map `reply.mealPlan` → `KitchenMealPlan` via `mapCaptainMealPlan` (adds a single `"Basic ingredients"` placeholder per meal, rebuilds day indices by grouping in 3s).
3. Else deterministic fallback (existing templates).

**Impact:** for the Arabic path that previously produced rich JSON via `CAPTAIN_ARABIC_API_URL`, users will now more frequently land on the placeholder-ingredient path. This is a quality regression but not a functional one — the plan still shows titles, types, and calories per day. Flagged for P0.3 as: "bring Kitchen-schema JSON parsing into `BrainOrchestrator` / `CloudBrainService` via a screen-context-specific output contract."
