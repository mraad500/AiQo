## Migration logic summary

- Added `MemorySchemaV4` as schema version `4.0.0` with the new memory primitives plus the preserved V3 models that still remain live in the store.
- Replaced the migration plan with a custom `CaptainSchemaV3 -> MemorySchemaV4` stage.
- `willMigrate` reads all legacy `CaptainMemory` and `PersistentChatMessage` rows, maps them into in-memory seed payloads, and logs staged counts without mutating source data.
- `didMigrate` inserts `SemanticFact` and `EpisodicEntry` rows from those seeds using deterministic IDs so the stage is idempotent if it is re-entered in-process.
- Legacy memory keys, raw categories, raw sources, chat `sessionID`, per-message IDs, captain reply timestamps, and Spotify recommendation payloads are preserved for compatibility with the existing app APIs.
- `MemoryStore` now supports both `.legacyV3` and `.schemaV4` backends behind the feature flag, so the current app still reads V3 while the V4 path is ready for rollout.
- `AiQoApp` now has explicit `makeCaptainContainerV3()` and `makeCaptainContainerV4()` boot paths, with `MEMORY_V4_ENABLED` gating the live schema selection.

## Known risks for production rollout

- `MEMORY_V4_ENABLED` is still `false`, so the migration path is staged but not active in normal app boots.
- Rolling a device back from an already-migrated V4 on-disk store to a V3 boot path is still operationally risky; the safe rollback story still depends on keeping the flag off in production until broader validation is complete.
- `MemoryBackup.exportV3Snapshot()` is intentionally still a placeholder and should be implemented before any production enablement if we want a user-restorable backup path.
- The build now emits Swift 6 isolation warnings around Codable conformances in the new snapshot/episode helpers. They do not fail the current build or tests, but they should be cleaned up before a stricter toolchain treats them as errors.
